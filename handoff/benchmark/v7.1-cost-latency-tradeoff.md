# AtlasPI v7.1 — Cost & Latency Tradeoff for Production Integration

**Audience**: AI agent developers integrating AtlasPI MCP server in production
**Status**: post-v7.0 benchmark closure (ADR-007)
**Last update**: 2026-04-23

---

## TL;DR

| Metric | Baseline (Claude alone) | AtlasPI full (with tools) | Multiplier |
|---|---|---|---|
| Latency end-to-end | ~3-5s | **~10-17s** | **~3×** |
| Output tokens (avg) | ~120 | ~280 | **~2×** |
| Input tokens (avg) | ~150 | ~6,000-12,000 | **~40-80×** |
| API cost/Q (Sonnet 4.5) | ~\$0.003 | **~\$0.020-0.040** | **~7-13×** |
| Hallucination rate | 0.21/Q | **0.07/Q** | **-67%** ✅ |
| Accuracy (hard Q) | 88.6% | **90.5%** | +2pp |

**Decision matrix for integrators**:

| Use case | AtlasPI worth it? | Reason |
|---|---|---|
| Mass-consumer Q&A (Wikipedia-level) | ❌ No | Ceiling effect — baseline already 92% |
| Research assistant (academic) | ✅ Yes | Hallucination reduction critical |
| Specialized historical agent | ✅ Yes | Tool-domain fit |
| Fast chatbot (<3s SLA) | ❌ No | Latency 3× too slow |
| Batch fact-checking | ✅ Yes | Throughput acceptable, accuracy primary |

## 1. Latency breakdown

### Baseline (Claude Sonnet 4.5, no tools)

```
User question → Claude API → Response
                 ~3-5s
```

Median measurement (from full-v1 results): **~4030 ms** per Q.

### AtlasPI full (Claude + 5 MCP tools)

```
User question → Claude API → tool_use signal
                 ~3-4s
                                ↓
                            HTTP call to AtlasPI REST → response back to Claude
                                ~0.5-1s per tool call (avg 2-4 tool calls)
                                ↓
                            Claude API → Response
                            ~3-5s
```

Median measurement (from full-v1 results): **~16,800 ms** per Q (~3-4× baseline).

**Why so much slower**:
1. Sequential tool loop: Claude must wait for HTTP response before deciding next action.
2. Each tool roundtrip: ~500-1000ms (Anthropic latency + AtlasPI server response).
3. Final LLM response after tool data gathered: another ~3-4s.
4. Tool call overhead: 2-4 typical, 6+ for complex queries (e.g. cross-entity comparison).

### Mitigation strategies

| Strategy | Latency impact | Trade-off |
|---|---|---|
| **Parallel tool calls** (Claude supports tool_use[] arrays) | -30% to -50% | Requires prompt engineering, less common |
| **Cache AtlasPI responses** (Redis at MCP layer) | -10% to -40% | Stale data risk on dynamic content |
| **Limit max_tool_iterations** (current default 8) | Bounds worst-case | May cut off legitimate exploration |
| **Streaming responses** | Same total but faster TTFB | Better UX, no compute reduction |
| **Smaller model for orchestration** (Haiku for tool decisions, Sonnet for synthesis) | -20% to -40% | Quality risk in routing decisions |

## 2. Token cost breakdown

### Baseline

- **Input tokens**: ~150 (system prompt + user question)
- **Output tokens**: ~120 (typical 2-4 sentence response)
- **Cost** (Sonnet 4.5: \$3/M input, \$15/M output):
  - input: \$0.00045
  - output: \$0.0018
  - **total: \$0.00225 per Q**

### AtlasPI full

Per Q with avg 3 tool calls:

- **Input tokens** ~6,000-12,000:
  - Initial: 250 (system prompt with tool definitions + question)
  - After each tool call, full conversation context is re-sent: 250 + 800 (tool response) = 1,050 per turn
  - 3 tool turns × 1,050 + final synthesis = ~6,000-12,000
- **Output tokens** ~280:
  - Tool decision JSON: 50-100 per tool call
  - Final response: 150-200
- **Cost**:
  - input: \$0.018-0.036
  - output: \$0.0042
  - **total: \$0.020-0.040 per Q (~10× baseline)**

### Per 1000 Q/day projection

| Mode | Daily cost | Monthly cost |
|---|---|---|
| Baseline | \$2.25 | \$67 |
| AtlasPI full | \$25-40 | \$750-1,200 |

For high-volume integrators, AtlasPI adds \$700-1,150/month per 1k Q/day. Worth it only if hallucination reduction has business value (e.g. academic publishing, professional research, regulated content).

### Cost mitigation

| Strategy | Cost reduction |
|---|---|
| **Prompt caching** (Anthropic feature, beta): system prompt + tool definitions cached | -50% to -70% on input tokens for repeated agents |
| **Slim tool responses**: AtlasPI atlaspi_agent.py already does this (omits boundary_geojson, sources, territory_changes from get_entity output) | -30% input on tool data |
| **Conditional tool use**: skip tools for clearly-easy questions (low-confidence routing) | Variable |
| **Batch endpoints** when known: `/v1/entities/batch?ids=1,2,3` instead of 3× `/v1/entities/{id}` | -50% latency on multi-entity queries |

## 3. Reliability tradeoffs

### Hallucination rate (vendable metric)

From bank v2 hard (30 Q designed to trap LLM baseline):

| Condition | Hallucinations/Q avg | vs Baseline |
|---|---|---|
| Baseline alone | 0.21 | — |
| Prompt-only (without tools) | 0.37 | **+76%** ⚠️ DAMAGING |
| **AtlasPI full** | **0.07** | **-67%** ✅ |

**Implication**: cost increase of 10× per Q is justified for use cases where ONE hallucination has cost > \$0.20. Examples:
- Academic publication errata
- Fact-checked content (newspapers, Wikipedia editing, encyclopedias)
- Regulated industries (legal, medical historical references)
- Long-context research where wrong fact propagates

### Reliability of tool calls

From benchmark observations:

- **Tool call success rate**: ~98% (rare 429 rate-limits, ~0% 5xx errors)
- **AtlasPI API uptime**: 99.5% measured (Aruba VPS + healthcheck)
- **Tool returns empty**: ~15% of calls (search misses, partial data); now handled by revised v7.1 prompt to defer to training

### Failure modes & mitigations

| Failure | Frequency | Mitigation in v7.1 prompt |
|---|---|---|
| Tool returns empty list | ~15% | Agent says "AtlasPI has no data for X, from training: ..." |
| Tool partial (e.g. 2 rulers when 10 expected) | ~10% | Agent supplements with training, marks gap explicitly |
| Tool timeout (>30s) | <0.1% | Bubble error to user, retry once |
| Rate limit (429) | ~0.5% | Backoff 10s, retry up to 3 times |

## 4. Decision matrix for integrators

### Use AtlasPI MCP if:

- Domain is historical geography / polities / rulers / events
- Hallucination cost > \$0.20 per error
- Latency SLA tolerates 10-15s per Q
- Volume <10k Q/day (cost-effective)
- User base = academic, professional, research-oriented

### Skip AtlasPI MCP if:

- Domain is general Q&A (Wikipedia coverage suffices)
- Real-time (<3s) chatbot
- Volume >100k Q/day (cost prohibitive without caching)
- User base = mass-consumer

### Alternative: AtlasPI as REST-only (no MCP)

If MCP integration overhead is too high, integrate via direct REST:

```python
import requests

def lookup_entity(query):
    r = requests.get(f"https://atlaspi.cra-srl.com/v1/search/fuzzy?q={query}")
    return r.json()

# Then pass result to LLM as context
```

This loses tool_use orchestration but adds AtlasPI data to RAG-style flow. Lower integration complexity, similar accuracy benefit on selected queries.

## 6. Reference numbers (raw from v7.0 benchmark)

From `scripts/benchmark/results/full-v1/raw_responses.jsonl` (100 Q, Claude Sonnet 4.5):

```python
import json, statistics
rows = [json.loads(l) for l in open('full-v1/raw_responses.jsonl')]

# Baseline metrics
b = [r['baseline'] for r in rows if isinstance(r['baseline'], dict) and r['baseline'].get('response')]
print(f"Baseline: latency median={statistics.median([x['latency_ms'] for x in b]):.0f}ms")
print(f"          tokens_out median={statistics.median([x['tokens_out'] for x in b]):.0f}")

# AtlasPI metrics
a = [r['atlaspi'] for r in rows if isinstance(r['atlaspi'], dict) and r['atlaspi'].get('response')]
print(f"AtlasPI:  latency median={statistics.median([x['latency_ms'] for x in a]):.0f}ms")
print(f"          tokens_out median={statistics.median([x['tokens_out'] for x in a]):.0f}")
print(f"          tool_calls median={statistics.median([x.get('tool_calls_count',0) for x in a]):.0f}")
```
